Introduction

This document explains the device structure and display driver entry points. It should contain enough information to enable users to create their own display driver. Broadcom specific enhancements from standard u8g interface are described in blue.

U8g library uses concept of pages to minimize size of frame buffer. Library reuses the same memory to construct different parts of screen and then prepared bytes are pushed to the display controller before buffer is reused to construct other part of the screen.  Details of pages and Picture Loop concept are discussed in this document.

To use optimized SPI driver, provided by Broadcom, size of memory needed for single page cannot exceed 1024 bytes. This limits horizontal resolution of the display to 512 pixels if two bytes are used for colors [65k colors], as in example display driver provided by Broadcom. If one byte is used for colors, horizontal resolution can be extended to 1024 pixels and so on. Higher resolution display can be used, but refreshing speed will be significantly affected and different version of SPI driver will be needed.

Drivers directory in apps\display\drivers directory contains all drivers from u8g project. One can use them as an example implementation to use for other controllers and displays as needed. In the same directory there are two drivers (with names ending _brcm) which were implemented by Broadcom to better utilize HW and SPI driver provided on 2073x chips.

To see Broadcom specific changes (enhancement) in display driver compare u8g_dev_ssd1351_128x128_hicolor_fn in u8g_dev_ssd1351_128x128.c with  u8g_dev_ssd1351_128x128_hicolor_65k_fn in u8g_dev_ssd1351_128x128_brcm.c

What is a U8glib device?

The U8glib device is a C structure:

struct _u8g_dev_t
{
  u8g_dev_fnptr dev_fn;         /* device procedure */
  void *dev_mem;                /* device memory */
  u8g_com_fnptr com_fn;         /* communication procedure */
};
typedef struct _u8g_dev_t u8g_dev_t;

There are three parts in a device structure:

  1. dev_fn:   A callback procedure, which handles messages from the graphics procedures.
  2. dev_mem: A pointer to some local memory for page buffer structure. If Broadcom communication driver is used this page memory cannot be bigger than 1024 bytes.
  3. com_fn:   A communication procedure

Display specific procedures (display init, pixel set and clear) are handled by dev_fn and communication specific details (SPI) are handled via com_fn.

It is possible that dev_fn directly writes to the display device. In this case set com_fn to u8g_com_null_fn.

Here is a template for a very simple device:

uint8_t my_dev_fn(u8g_t *u8g, u8g_dev_t *dev, uint8_t msg, void *arg)
{
  /* handle messages here */
  return 1; /* successful handled */
}

u8g_dev_t u8g_dev_my_simple_display = { my_dev_fn, NULL, u8g_com_null_fn };

 

To connect display driver with communication driver and page buffer, u8g_pb_t and u8g_dev_t structures are used. They are defined at the end of display driver file.

Messages (entry points) to the display drivers

All the communication between the graphics procedures and the display driver callback procedure (dev_fn) is done via messages. A message is a number, which is passed in the argument msg of the dev_fn:

uint8_t my_dev_fn(u8g_t *u8g, u8g_dev_t *dev, uint8_t msg, void *arg)

In some cases, there are additional arguments and parameters together with msg. In this case arg is a none-NULL pointer to these additional arguments.

A typical device callback will look like this:

uint8_t my_dev_fn(u8g_t *u8g, u8g_dev_t *dev, uint8_t msg, void *arg)
{
  switch(msg)
  {
    case U8G_DEV_MSG_INIT:
      /* setup/init display */
      /* ... */
      break;

    case U8G_DEV_MSG_PAGE_NEXT:
      /* write a page to the display */
      /* ... */
      break;

    /* maybe handle more messages */
  }
  /* make a call to a suitable page buffer procedure   [in this example: 8 bits height vertical orientation of byte (1 bit color)] */
  return u8g_dev_pb8v1_base_fn(u8g, dev, msg, arg);
}

This is a reference to all of messages:

  • U8G_DEV_MSG_INIT: Setup and init display, no args. If arg is not null, the provided pointer should be set to SPI FIFO address (*arg= &FIFO).
  • U8G_DEV_MSG_STOP: Shutdown display, no args (not used at the moment).
  • U8G_DEV_MSG_CONTRAST: Set contrast or brightness of the display, arg is a pointer to uint8_t which contains the contrast value (0..255).
  • U8G_DEV_MSG_SLEEP_ON: Display should enter sleep mode, no args.
  • U8G_DEV_MSG_SLEEP_OFF: Display should recover from sleep mode, no args.
  • U8G_DEV_MSG_PAGE_FIRST: Picture loop will start now. Page buffer procedures will reset the page handling to the first page and clear the buffer.
  • U8G_DEV_MSG_PAGE_NEXT: A page has been finished, dev_fn should transfer page to the device, page buffer procedures will setup next page. 0 should be returned if there are no further pages required. There are no args for this message.
  • U8G_DEV_MSG_SET_PAGE: Set next page to be refreshed. Allows skipping of pages which do not need refreshing. 
  • U8G_DEV_MSG_GET_PAGE_BOX: Return dimensions of the page. arg is a pointer to u8g_dev_arg_bbx_t. This message is handled by page buffer procedures.
  • U8G_DEV_MSG_SET_PIXEL: Set a pixel. arg is a pointer to u8g_dev_arg_pixel_t. This message is handled by page buffer procedures.
  • U8G_DEV_MSG_SET_8PIXEL: Set a sequence of 8 pixel. arg is a pointer to u8g_dev_arg_pixel_t. This message is handled by page buffer procedures.
  • U8G_DEV_MSG_SET_COLOR_INDEX (until v1.12): Not used
  • U8G_DEV_MSG_SET_COLOR_ENTRY (since v1.13): Set color palette entry. arg is a pointer to u8g_dev_arg_irgb_t.
  • U8G_DEV_MSG_SET_XY_CB: Not used
  • U8G_DEV_MSG_GET_WIDTH: Return the width of the display. arg is a pointer to u8g_uint_t. The width must be stored at the address pointed to by arg.
  • U8G_DEV_MSG_GET_HEIGHT: Return the height of the display. arg is a pointer to u8g_uint_t. The height must be stored at the address pointed to by arg.
  • U8G_DEV_MSG_GET_MODE: Return the display mode as return value of dev_fn, no other args. Possible return values are U8G_MODE_BW and U8G_MODE_GRAY2BIT

 

Messages (entry points) to the communication drivers

All the communication between the display driver (dev_fn)  and communication driver (com_fn) is done via messages. A message is a number, which is passed in the argument msg of the com_fn:

uint8_t my_com_fn(void *u8g, uint8_t msg, uint32_t arg_val, void *arg_ptr)

There are two additional arguments passed together with msg. In most cases only one of them is used and communication callback knows which one to expect based on the msg.

Basically: if single variable is passed arg_val is used, if multiple variables are needed (or block of data) arg_ptr is used.

This is a reference to all of messages and its arguments:

  • U8G_COM_MSG_INIT: Setup and init display, no args. If arg_ptr is not null, the provided pointer will be set to SPI FIFO address (*arg= &FIFO) by Broadcom  SPI driver.
  • U8G_COM_MSG_STOP: Shut down, no args (not used at the moment).
  • U8G_COM_MSG_ADDRESS: Change communication mode to cmd (arg_val == 0) or data (arg_val ==1).
  • U8G_COM_MSG_CHIP_SELECT: Enable SPI chip select line if arg_val == 1. Not used by Broadcom driver as HW is providing auto_cs functionality. Kept for compatibility with other drivers.
  • U8G_COM_MSG_RESET: Set display reset line low or high.  Arg_val provides reset line state.
  • U8G_COM_MSG_WRITE_BYTE: Write single byte. Arg_val has the byte.
  • U8G_COM_MSG_WRITE_SEQ: Write sequence of bytes from ROM or RAM. Arg_val has the bytes length. Arg_ptr points to the buffer with bytes to be written.

·         U8G_COM_MSG_WRITE_SEQ_P: Write sequence of bytes from EEPROM. Not supported in BRCM driver as 20739 does not support EEPROM.
